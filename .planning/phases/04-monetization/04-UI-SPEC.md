# Phase 4 — UI Design Contract

> Visual and interaction contract for Phase 4: Monetization (Interstitial Ads, Rewarded Video, Pro IAP, Restore Purchases).
> Generated by gsd-ui-researcher.
> Based on decisions D-87 through D-112 (04-CONTEXT.md), 04-RESEARCH.md, and upstream design tokens from Phases 1-3 UI-SPEC.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (React Native — no shadcn/tailwind) |
| Preset | not applicable |
| Component library | Built-in RN components + react-native-google-mobile-ads (AdMob) + react-native-iap (Play Store) |
| Icon library | `@expo/vector-icons` (MaterialIcons) — already used for home button in ResultModal |

### 3. Restore & Purchase Flow — `src/screens/SettingsScreen.tsx`

**Contract:** SettingsScreen gains restore/purchase logic with toast feedback.

#### Restore Flow (D-99 through D-102)

| Step | UI Behavior |
|------|-------------|
| 1. User taps "Restore Purchases" | RestoreRow shows `ActivityIndicator` on right (loading state) |
| 2. `RNIap.getAvailablePurchases()` resolves | Loading indicator replaced by result |
| 3a. Valid Pro purchase found | `settingsStore.setPro(true)` → row hides (filtered). Green toast: "Pro restored!" (3s auto-dismiss) |
| 3b. No purchase found | Red toast: "No purchases found to restore" (3s auto-dismiss). Row remains visible. |
| 4. Network/API error | Red toast: "No purchases found to restore" (3s auto-dismiss). Row remains visible. |

#### Purchase Flow (D-95 through D-98)

| Step | UI Behavior |
|------|-------------|
| 1. User taps "Remove Ads · $1.99" | PurchaseRow shows `ActivityIndicator` on right (loading state). Then native Play Store billing dialog opens (no custom UI). |
| 2a. Purchase successful via `purchaseUpdatedListener` | `finishTransaction` → `settingsStore.setPro(true)` → purchase row hides (filtered). Green toast: "Welcome to Pro!" (3s auto-dismiss) |
| 2b. Purchase cancelled by user | Loading indicator removed. No toast. Row remains visible. |
| 2c. Purchase error | Red toast: "Purchase failed. Please try again." (3s auto-dismiss). Row remains visible. |


#### Visual Contract

| Aspect | Value |
|--------|-------|
| Component | `TouchableOpacity` with `activeOpacity: 0.7` |
| Background | `colors.accent` (#4a9eff) |
| Text | "Watch Ad for +1 Guess" — 16px, 600 weight, `colors.textInverse` (#ffffff) |
| Border radius | 12px |
| Padding | vertical: 14px, horizontal: 24px |
| Width | 100% (full card width) |
| Margin top | 8px (gap from button container above) |

#### State Machine

```
        ┌─────────┐
        │  HIDDEN  │ ← isPro? no... wait, Pro users CAN see this per D-93
        └────┬─────┘
             │ (session.status === 'lost' && extraGuessesUsed < maxExtra)
             ▼
        ┌─────────┐    tap     ┌────────────┐   ad ready   ┌───────────┐
        │ VISIBLE │ ────────→ │ SHOWING AD  │ ──────────→ │ AD PLAYED │
        └─────────┘           └─────────────┘              └─────┬─────┘
             │                        │                          │
             │                        │ (ad not loaded)          │ (onRewarded)
             │                        ▼                          ▼
             │                  ┌──────────────┐          ┌──────────────┐
             │                  │ FALLBACK:    │          │ EXTRA GUESS  │
             │                  │ do nothing   │          │ GRANTED      │
             │                  │ (stay visible)│         │ (addExtraGuess)
             │                  └──────────────┘          │ modal closes  │
             │                                            │ game resumes  │
             │                                            └──────────────┘
             │
             │ (extraGuessesUsed >= maxExtra)
             ▼
        ┌─────────┐
        │  HIDDEN  │
        └─────────┘
```

**Show conditions** (D-89, D-92, D-93, D-94):
- `session.status === 'lost'` (loss state only)
- `session.extraGuessesUsed < maxExtra` where `maxExtra = isPro ? maxExtraGuessesPro(3) : maxExtraGuessesFree(2)`
- Pro users see the button (D-93: Pro removes interstitials only, not rewarded ads)

**On press:**
1. Call `adStore.showRewarded(onRewarded: callback)`
2. If `showRewarded` returns `false` (ad not loaded): do nothing. Button stays visible. User can try again.
3. If ad shown → on EARNED_REWARD event: callback fires → `gameStore.addExtraGuess()` → `session.status` changes from 'lost' to 'playing' → `maxAttempts++`, `extraGuessesUsed++` → ResultModal closes (since session is no longer terminal) → game resumes with board showing previous guesses + new empty row.

### 5. Interstitial Ad Transition — `src/components/game/ResultModal.tsx`

**Contract:** Interstitial fires BEFORE navigation on "Back to Menu" / "Play Next" tap, gated by frequency and Pro status.

#### Trigger Flow

```
User taps "Play Next" or "Back to Menu"
         │
         ▼
  ┌─────────────────┐
  │ isPro?          │──yes──→ Navigate immediately (no ad)
  └────────┬────────┘
           │ no
           ▼
  ┌─────────────────┐
  │ Frequency check  │
  │ gamesSinceLastAd │
  │ >= threshold?    │──no──→ Navigate immediately
  └────────┬────────┘
           │ yes
           ▼
  ┌─────────────────┐
  │ Show interstitial│
  │ (await close)    │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ Reset frequency  │
  │ counter          │
  └────────┬────────┘
           │
           ▼
      Navigate
```

#### Frequency Thresholds (D-88)

| Mode | Threshold (`gamesSinceLastAd >= N`) |
|------|-------------------------------------|
| Daily | 0 (every game) |
| Endless | 1 (every 2nd game) |
| Random | 1 (every 2nd game) |

**Implementation:** `gamesSinceLastAd` is incremented by `GameScreen` on game completion (before modal shows). At navigation time, threshold check compares current counter value. After interstitial shown, counter reset via `adStore.resetGamesSinceLastAd()`.

#### Edge Cases

| Scenario | Behavior |
|----------|----------|
| Interstitial not loaded | Skip ad, navigate immediately. Ad store `showInterstitial()` returns false. |
| Interstitial closed by user | Proceed with navigation. Counter reset (ad was shown). |
| Interstitial load error | Skip ad, navigate immediately. No error UI needed. |
| Pro user | Skip entire interstitial check (D-93, AD-05). Navigate immediately. |
| Rewarded ad in progress | Prevent interstitial during rewarded ad (both can't overlap). Ad store manages this. |

### 6. GameScreen — Interstitial Preload & Frequency Counter

**Contract:** `src/screens/GameScreen.tsx` integrates ad lifecycle hooks.

#### Preload at Game Start

When a game starts (in the existing game initialization `useEffect`):
```
adStore.preloadInterstitial()
```
This loads the interstitial in the background so it's ready when the game ends.

#### Preload After Game Ends

When game completes (win or loss), after animation finishes and stats recording:
```
adStore.incrementGamesSinceLastAd()  // increment frequency counter
adStore.preloadInterstitial()         // preload next ad for next game
```

#### No Loading Indicator

Interstitial preloading has no visible UI — it happens silently in the background. The `adStore` state tracks `interstitialLoaded` internally; no spinner or toast is shown during loading.

### 7. GameStore — addExtraGuess() Action

**Contract:** New action method on gameStore (not a UI component, but defines interaction boundary).

| Property | Contract |
|----------|----------|
| Precondition | `session.status === 'lost'` — only works from loss state (D-89) |
| Postcondition | `session.status = 'playing'`, `session.maxAttempts += 1`, `session.extraGuessesUsed += 1`, `currentGuess = ''`, `error = null` |
| Cap enforcement | Checks `extraGuessesUsed < maxExtra` per tier (D-92, D-94). If at cap, no-op. |
| Pro cap | `isPro ? config.maxExtraGuessesPro(3) : config.maxExtraGuessesFree(2)` |
| Return value | `void` — UI observes state change via Zustand subscription |

## Edge Cases

### Settings Account Section

| Scenario | Behavior |
|----------|----------|
| **First launch (isPro=false)** | Account section shows all 4 rows: sign-in placeholder, pro status "—", purchase row, restore row |
| **Pro purchased this session** | Purchase row and restore row hide immediately after `setPro(true)`. Toast shows "Welcome to Pro!". Pro status value updates to "Active" (green). |
| **Pro restored from previous purchase** | Same as above — rows hide, toast shows "Pro restored!" |
| **Fresh install on new device (no purchase history)** | Purchase and restore rows visible. Restore returns "No purchases found to restore" (red toast). Purchase row opens native Play Store dialog. |
| **App data cleared (MMKV wiped)** | `isPro` resets to `false`. Purchase and restore rows re-appear. User can restore from purchase history. (D-102) |
| **Restore while offline** | `getAvailablePurchases()` throws network error. Caught → red toast. Row remains visible. |
| **Purchase while offline** | `requestPurchase()` needs network — Play Store handles this natively (shows error dialog or fails). Caught → red toast. |
| **Rapid restore taps** | Debounce: restore is async; loading state prevents double-tap (ActivityIndicator shown while promise pending) |
| **Rapid purchase taps** | Same debounce via loading state |
| **Sign-in placeholder coexistence** | Placeholder row remains unchanged. (D-111) |

### ResultModal — Rewarded Ad Button

| Scenario | Behavior |
|----------|----------|
| **Win state** | Button not shown (D-89: loss state only) |
| **Loss state, extra guesses remaining** | Button visible, functional |
| **Loss state, at max extra guesses** | Button hidden (`extraGuessesUsed >= maxExtra`) |
| **Pro user, loss state, extra guesses remaining** | Button visible (D-93: Pro can still watch rewarded ads). Max cap = 3 instead of 2. |
| **Pro user, at max extra guesses** | Button hidden |
| **Rewarded ad not loaded** | Button visible but tap does nothing (silent no-op). User can try again. |
| **Rewarded ad loads mid-display** | No change — ad is ready for next tap. |
| **Extra guess limit reached mid-game** | Not applicable — limit checked in addExtraGuess() and in button visibility |
| **Network drops during rewarded ad** | Ad SDK handles this natively. Button remains visible for retry. |
| **User dismisses rewarded ad before reward** | No reward granted. Button remains visible. User can try again. |
| **Endless mode loss with extra guess** | Button shown. On extra guess granted, modal closes, game resumes with new attempt. Next game completion increments `gamesSinceLastAd`. |
| **Daily mode loss with extra guess** | Same flow as Endless — game resumes mid-session. |

### Interstitial Transitions

| Scenario | Behavior |
|----------|----------|
| **Pro user taps "Play Next"** | No interstitial check. Navigates immediately. (D-93) |
| **Free user, Daily mode, taps "Play Next"** | Interstitial shown (threshold 0). After ad closes, next game starts. |
| **Free user, Endless mode, first game just completed** | `gamesSinceLastAd` = 0. Threshold = 1. No interstitial. Navigates immediately. Counter unchanged. |
| **Free user, Endless mode, second game just completed** | `gamesSinceLastAd` = 1. Threshold = 1. Interstitial shown. Counter reset to 0. |
| **Interstitial fails to load** | `showInterstitial()` returns false. Navigate immediately. Counter NOT reset (so it will try again next transition). |
| **Interstitial loads but user dismisses quickly** | Counter reset. Ad was shown. Next game/future transition will check counter. |
| **Free user, no games played yet this session** | `gamesSinceLastAd` = 0. No interstitial on first game completion navigation in Endless/Random mode (threshold 1). Daily mode shows interstitial. |
| **App backgrounded during interstitial** | Ad SDK handles this natively. When app returns: if interstitial was dismissed, navigation proceeds. |

### General

| Scenario | Behavior |
|----------|----------|
| **Toast auto-dismiss interrupted by navigation** | Toast is local state to SettingsScreen. If user navigates away before 3s, toast disappears when component unmounts. Acceptable for MVP. |
| **Multiple toasts overlapping** | Each new toast replaces the previous (setState overwrite). `clearTimeout` on unmount to prevent memory leak. |
| **Back button during interstitial** | Android `BackHandler` — interstitial is fullscreen native overlay; back button behavior is handled by AdMob SDK (typically dismisses ad). After ad: navigation proceeds. Phase 6 will add explicit back-button blocking. |
| **Back button during purchase flow** | Play Store billing dialog handles back button natively. |
