<task type="auto">
  <name>Task 2: Create Firestore service for player stats sync and leaderboard data</name>
  <files>src/services/firestoreService.ts</files>
  <action>
    Create `src/services/firestoreService.ts` — a pure TypeScript module wrapping @react-native-firebase/firestore.

    **Design requirements:**
    - D-128 through D-132 (Firestore collection structure, merge strategy, query limits)
    - Idempotent writes using `set({...}, { merge: true })`
    - Graceful fallback when Firebase is not configured (no crash)
    - Event-based incremental sync: each game writes an atomic update (not full-state overwrite)

    **Exported functions:**

    ```typescript
    import firestore from '@react-native-firebase/firestore';
    import type { PlayerStats, LeaderboardEntry, LeaderboardData } from '../types';

    type LeaderboardType = 'daily_streak' | 'endless_streak' | 'endless_total';

    /**
     * Update a player's stats document in Firestore.
     * Uses { merge: true } for idempotent incremental updates (D-131).
     * Returns false if Firestore is not available (no crash).
     */
    async function updatePlayerStats(
      playerId: string,
      playerName: string,
      stats: PlayerStats
    ): Promise<boolean>;

    /**
     * Fetch a player's stats document from Firestore.
     * Returns null if not found or if Firestore is unavailable.
     */
    async function getPlayerStats(playerId: string): Promise<PlayerStats | null>;

    /**
     * Submit or update a leaderboard score for a player.
     * Uses { merge: true } for upsert — duplicate game completions overwrite with latest score.
     * Collection path: leaderboards/{type}/scores/{playerId}
     * Document fields: { playerId, playerName, score, updatedAt }
     * Returns false on failure (network error, not configured).
     */
    async function submitLeaderboardScore(
      type: LeaderboardType,
      playerId: string,
      playerName: string,
      score: number
    ): Promise<boolean>;

    /**
     * Query top 50 entries from a leaderboard collection.
     * Ordered by score descending, limited to 50 (D-132).
     * Returns LeaderboardData with entries array, or falls back to empty array on error.
     */
    async function getLeaderboard(type: LeaderboardType): Promise<LeaderboardData>;
    ```

    **Implementation details:**
    - Collection path convention: `leaderboards/{type}/scores/{playerId}` — this is a Firestore collection group pattern that allows querying across all types or per type
    - Actually simpler: use subcollections under a single leaderboards document
    - Path: `leaderboards/{type}/scores/{playerId}` where type is `daily_streak`, `endless_streak`, or `endless_total`
    - Each score document: `{ playerId: string, playerName: string, score: number, updatedAt: Timestamp }`
    - Query: `firestore().collection('leaderboards').doc(type).collection('scores').orderBy('score', 'desc').limit(50)`
    - Wrap all Firestore calls in try/catch — if Firebase is not configured or network fails, return fallback values (false, null, empty array)
    - Use `firestore().settings({ persistence: true })` for Firestore native offline cache (supplements our explicit sync queue)

    **State management notes:**
    - This is a stateless service module, not a Zustand store
    - Consumers (statsStore, LeaderboardScreen) import and call functions directly
    - No internal caching — let Firestore SDK handle local cache via persistence

    **Error handling strategy:**
    - All functions catch errors and return typed fallback values
    - `submitLeaderboardScore` returns false on failure (caller decides whether to queue)
    - No console.error noise for expected offline scenarios
    - Firestore internal retry handles transient network errors; our syncQueue handles persistent offline cases
  </action>

<task type="auto">
  <name>Task 3: Create offline sync queue with idempotent events, retry, and exponential backoff</name>
  <files>src/services/syncQueue.ts</files>
  <action>
    Create `src/services/syncQueue.ts` — an AsyncStorage-backed write-ahead log for offline game results and deferred score submissions.

    **Design requirements:**
    - D-133 through D-140 (AsyncStorage queue, event types, idempotent IDs, exponential backoff, retry limits)
    - Must handle 3+ offline games without data loss (unbounded queue size)
    - Event IDs are deterministic (SHA-256 hash of session ID + event type) for idempotent dedup
    - Exponential backoff: `2^retryCount * 1000ms` — retries at 1s, 2s, 4s (max 3 retries)
    - Events with retryCount >= 3 and nextRetryAt in the past are discarded on drain attempt
    - Must not block the main thread — all operations are async
    - No external dependencies beyond AsyncStorage (already in project)

    **Type definitions:**

    ```typescript
    interface SyncEvent {
      id: string;           // deterministic hash for idempotency
      type: 'game_result' | 'leaderboard_score';
      data: Record<string, unknown>;
      retryCount: number;
      nextRetryAt: string | null;  // ISO timestamp, null = can retry now
      createdAt: string;           // ISO timestamp
    }
    ```

    **Exported functions:**

    ```typescript
    /**
     * Enqueue a sync event to the offline queue.
     * If an event with the same ID already exists, skip (idempotent) — D-136.
     * Returns true if enqueued, false if duplicate skipped.
     */
    async function enqueueEvent(
      type: SyncEvent['type'],
      data: Record<string, unknown>,
      id?: string  // optional explicit ID; auto-generated from hash if omitted
    ): Promise<boolean>;

    /**
     * Drain the queue: iterate through all pending events, call handler for each.
     * - On handler success: remove event from queue.
     * - On handler failure: increment retryCount, set nextRetryAt = now + 2^retryCount*1000ms.
     * - Discard events with retryCount >= 3 (max retries exhausted) — D-148.
     * - If an event's nextRetryAt is in the future, skip it this pass (not ready yet).
     * Returns number of events processed (success or discarded).
     */
    async function drainQueue(
      handler: (event: SyncEvent) => Promise<boolean>
    ): Promise<number>;

    /**
     * Get the number of pending events in the queue.
     */
    async function getQueueLength(): Promise<number>;

    /**
     * Clear all events from the queue (for testing/admin).
     */
    async function clearQueue(): Promise<void>;
    ```

    **Implementation notes:**
    - AsyncStorage key: `'sync_queue'` — stores JSON array of SyncEvent objects
    - Read entire array on each operation (queue is small; AsyncStorage is KV-based, not a streaming DB)
    - Use `JSON.parse()`/`JSON.stringify()` for serialization
    - `enqueueEvent` checks for duplicate ID before pushing
    - `drainQueue` iterates from index 0, calls handler for each non-expired event
    - After processing, write back the updated queue (remaining events)
    - Event ID generation: use a simple hash (`djb2` or `fnv1a`-like) of `${sessionId}:${eventType}` — not crypto (not needed for idempotency)
    - Wrap all AsyncStorage operations in try/catch

    **Integration notes:**
    - This service is consumed by authStore (drain on sign-in) and GameScreen (enqueue on game completion)
    - The handler passed to drainQueue should be a function that calls firestoreService methods
    - Draining is non-blocking (fire-and-forget promise) — never await drainQueue in UI paths
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local app ↔ AsyncStorage | Local storage — no external trust boundary |
| App ↔ Firebase Firestore | Network boundary — Firestore rules enforce auth-based access |
| App ↔ Google Sign-In SDK | OAuth trust boundary — Google SDK handles token exchange |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-ID-01 | Spoofing | Firestore writes | Mitigated | Firestore Security Rules require authenticated writes (user.uid == playerId). Only game data, no PII. |
| T-ID-02 | Tampering | Sync queue | Accepted | Queue stored in AsyncStorage (app-private storage). Not user-accessible on non-rooted device. Events only carry game data, no auth tokens. |
| T-ID-03 | Information Disclosure | Leaderboard data | Accepted | Leaderboard entries are public (player name + score). No PII stored. Player IDs are Firebase UIDs (not email). |
| T-ID-04 | Denial of Service | Sync queue overflow | Mitigated | Unbounded queue size accepted (local storage). AsyncStorage limit is ~6MB per key. Each event ~500 bytes → ~12,000 events before limit. |
</threat_model>
